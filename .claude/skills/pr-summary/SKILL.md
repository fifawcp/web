# PR summary

## Goal

Open a PR whose body a reviewer can scan in under 30 seconds. Optimise for **signal density**, not coverage.

## Steps

1. **Gather context** in parallel:
   - `git log develop..HEAD --oneline` ÔÇö list of commits on the branch
   - `git diff develop...HEAD --stat` ÔÇö file-level overview
   - Current branch name (extract issue number if it matches `[A-Z]+-\d+`, e.g. `WEB-24` ÔåÆ `Closes #24`)
   - Read `.github/pull_request_template.md` to confirm section headers
   - Read enough of `git diff develop...HEAD` to understand intent ÔÇö chunk if large; you do **not** need to read every hunk

2. **Draft the PR body** following the template exactly (three sections, no extras):

   ```md
   ## Related issue

   Closes #<number>

   ## What changed

   - <bullet>
   - <bullet>

   ## How to test

   - [ ] <step>
   - [ ] <step>
   ```

3. **Show the draft to the user** as a fenced markdown block. Ask: *"Open the PR with this body, or want me to adjust?"* Wait for confirmation.

4. **On approval**, run:
   ```bash
   gh pr create --base develop --title "<title>" --body "$(cat <<'EOF'
   <body>
   EOF
   )"
   ```
   Use the most recent commit subject as the title unless the user provided one.

5. **Return the PR URL.** Nothing else.

## Output rules ÔÇö read carefully

These exist because past attempts produced 600-word PR bodies nobody read.

- **What changed: 3ÔÇô6 bullets total.** One line each. Group by area only if there are clearly distinct areas (feature / infra / i18n). No nested bullets. No section sub-headers inside.
- **How to test: 3ÔÇô6 checklist items.** Each one is a single concrete action a reviewer takes (`Visit /schedule and click X`, not `Verify the schedule works`). Cover the golden path + any auth/guest split if the route is shared. Skip exhaustive edge cases.
- **No preamble, no closing summary, no emoji.** No "This PR introducesÔÇª" framing. Start with the bullet.
- **Reference files only when it sharpens a bullet** ÔÇö `src/proxy.ts` is fine, `src/features/schedule/components/MatchCard.tsx` for "added MatchCard" is noise.
- **Skip the obvious**: don't bullet `package-lock.json`, formatter changes, or autogenerated files.
- **Body cap: ~150 words.** If it's longer, you're listing instead of summarising ÔÇö cut.

## Title rules

- Conventional Commits prefix (`feat`, `fix`, `chore`, `refactor`, `docs`).
- Under 70 chars.
- Use the dominant theme of the branch, not the latest commit if it's a small fixup.

## When to ask before drafting

- If the diff spans multiple unrelated themes, ask the user which is the PR scope before drafting ÔÇö don't try to summarise everything.
- If there's no issue number derivable from the branch or commits, ask for one rather than guessing.
